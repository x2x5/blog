Last winter, while tidying up my bookshelf, I found an old notebook with curled corners. On the first page, a line I had written in college:

> Keep writing, keep thinking — whether anyone reads it or not.

That was a message from my younger self. The handwriting was a bit immature, but the meaning was clear. I had blogged on various platforms back then — then work got busy, domains expired, articles scattered, and I stopped. This year I decided to start again, on a site I fully control.

## Choosing the Tech Stack

There are plenty of options for building a blog today. I tried a few before settling on Docusaurus:

| Solution | Type | Writing Experience | Flexibility | Maintenance |
|----------|------|-------------------|-------------|-------------|
| Docusaurus | SSG (React) | MDX, embeddable components | High | Low |
| Hugo | SSG (Go) | Markdown, template syntax | Medium | Low |
| Next.js + MDX | SSR/SSG | Fully customizable | Very High | High |
| WordPress | Dynamic | Visual editor | Medium | High |

What sold me on Docusaurus was its MDX support — I can use React components directly in articles, which is handy for technical posts with interactive elements. It also ships with i18n, search, RSS, and a clean theme out of the box.

The core config ended up looking like this:

```ts
// docusaurus.config.ts
const config: Config = {
  url: 'https://x2x5.top',
  baseUrl: '/blog/',
  i18n: {
    defaultLocale: 'zh-CN',
    locales: ['zh-CN', 'en'],
  },
  presets: [
    [
      'classic',
      {
        docs: false,
        blog: {
          routeBasePath: '/',
          remarkPlugins: [remarkMath],
          rehypePlugins: [rehypeKatex],
        },
      },
    ],
  ],
};
```

The key detail is `routeBasePath: '/'` — without it, the blog index would end up at `blog/blog/` instead of `blog/`. Chinese and English articles live in separate directories, and Docusaurus pairs them automatically by matching folder names.

## Writing Workflow

The biggest barrier to writing has always been getting into the right state of mind. Here are the rules I set for myself:

1. **Capture immediately** — jot down a title and a few lines in my phone's notes app
2. **Weekly refinement** — turn fragments into paragraphs in `blog/YYYY-MM-DD-slug/index.mdx`
3. **Ship imperfect** — publish the first draft, allow edits later
4. **Bilingual sync** — place the English version at the same path under `i18n/en/`

```bash
# Scaffold a new post
npm run new-post "Post Title" -- --slug post-slug
```

This command generates both the Chinese and English template files, saving the hassle of manual setup.

## On Math Formulas

Math rendering was a nice bonus in choosing Docusaurus. I studied information theory in college, and the Fourier transform has always fascinated me:

$$
F(\omega) = \int_{-\infty}^{\infty} f(t) e^{-i \omega t} \, dt
$$

The time domain and frequency domain are two ways of seeing the same signal — by changing your perspective, hidden patterns emerge. Writing a blog is similar: turning thoughts into words is like transforming from time to frequency domain. Things that felt blurry suddenly snap into focus.

Inline math works too, like $E = mc^2$ and $x^2 + y^2 = z^2$.

## Random Thoughts

A few things I learned while setting this up:

- **Reduce friction** — the more steps between you and publishing, the less you publish. I set up deployment, comments, and images once, so I only need to focus on content
- **Keep it simple** — no fancy animations, no complex taxonomies. Posts, tags, archive. That's enough
- **Own your data** — everything is Markdown files. I can move to any platform anytime

These principles apply beyond blogging. For any personal project, reducing friction and keeping things simple go a long way.
